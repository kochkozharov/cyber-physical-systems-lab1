# Лабораторная работа 1: CV

Вариант задания: **«на тройку»** (обнаружение и распознавание объектов, ultralytics / YOLOv11).

## 1. Датасет и обоснование выбора

**[African Wildlife Dataset](https://docs.ultralytics.com/datasets/detect/african-wildlife/)** — публичный датасет для обнаружения четырёх видов африканских животных на фотографиях: **буйвол, слон, носорог, зебра**. Архив содержит 1504 изображения уже разбитые в готовую YOLO-разметку:

| Split | Images |
|---|---|
| train | 1052 |
| val | 225 |
| test | 227 |

**Практическая задача:** автоматический мониторинг миграций и подсчёт популяции диких животных по снимкам с камер-ловушек и беспилотников в заповедниках. Это стандартная задача conservation ML — единое решение обнаруживает несколько таксонов в одной модели.

## 2. Метрики качества

| Метрика | Обоснование |
|---|---|
| **mAP@0.5** | Основная стандартная метрика детекции: площадь под кривой precision-recall при IoU ≥ 0.5. Именно она репортится в карточках моделей Ultralytics. |
| **mAP@0.5:0.95** | Строгая метрика COCO, усредняющая результаты по порогам IoU от 0.5 до 0.95 с шагом 0.05. Штрафует за неточную локализацию рамки. |
| **Precision** | Доля предсказанных рамок, которые верны. Важна, чтобы ренжеры не тратили время на ложные срабатывания. |
| **Recall** | Доля реальных животных, которые были найдены. Важна, чтобы не пропустить особь при учёте популяции. |

Для кастомного детектора (раздел 4) mAP@0.5 считается как доля изображений, где единственное предсказание модели совпало хотя бы с одной GT-рамкой (IoU ≥ 0.5 + верный класс).

## 3. Установка

Требуется **Python 3.12+** (проверено на Python 3.14, macOS arm64 / MPS). Для apple silicon PyTorch использует MPS; в скриптах выставлен `PYTORCH_ENABLE_MPS_FALLBACK=1` автоматически.

```bash
python3 -m venv .venv
source .venv/bin/activate     # bash / zsh
pip install -r requirements.txt
```

Датасет **скачивается автоматически** при первом запуске `prepare_dataset.py`. Архив (~105 MB) берётся из официального GitHub-релиза Ultralytics.

## 4. Скрипты

### `prepare_dataset.py`

Скачивает архив `african-wildlife.zip` из Ultralytics assets, распаковывает его в `dataset/african-wildlife/` (`images/{train,val,test}` + `labels/{train,val,test}`) и записывает `data/dataset.yaml` с абсолютными путями.

```bash
python prepare_dataset.py
```

Ожидаемый вывод:

```
Downloading dataset from https://github.com/ultralytics/assets/releases/download/v0.0.0/african-wildlife.zip
Extracting into .../dataset
Dataset prepared at .../dataset/african-wildlife
Wrote config to .../data/dataset.yaml
Dataset splits: 1052 train / 225 val / 227 test
```

### `train_baseline.py`

Обучает две базовые модели из семейства Ultralytics:

- **yolo11n** — YOLOv11 nano, чисто свёрточный детектор (~2.6 M параметров).
- **rtdetr-l** — RT-DETR large, трансформер-encoder/decoder (~32 M параметров).

Обе модели — **5 эпох, batch=16, imgsz=640, seed=42**, все остальные гиперпараметры дефолтные.

```bash
python train_baseline.py
```

Веса: `runs/baseline/{yolo11n,rtdetr_l}/weights/best.pt`.

### `train_improved.py`

Переобучает обе модели с улучшенной конфигурацией, проверяющей три гипотезы:

| Гипотеза | Изменение | Ожидаемый эффект |
|---|---|---|
| Более сильная аугментация | mosaic=1.0, mixup=0.15, HSV-джиттер, поворот ±5°, отражение | Лучшая обобщающая способность |
| Косинусный LR + прогрев | cos_lr=True, warmup_epochs=2 | Более стабильная сходимость |
| Регуляризация | weight_decay=5e-4, label_smoothing=0.05 | Снижение переобучения |

Бюджет увеличен до **15 эпох**, чтобы косинусный график и аугментации успели реально повлиять — на 5 эпохах warmup занимал бы 40% обучения.

```bash
python train_improved.py
```

Веса: `runs/improved/{yolo11n,rtdetr_l}/weights/best.pt`.

### `evaluate.py`

Оценивает сохранённые веса на **тестовой выборке** (`split=test`) и печатает markdown-таблицу с Precision / Recall / mAP@0.5 / mAP@0.5:0.95.

```bash
python evaluate.py                                        # все доступные модели
python evaluate.py yolo11n_baseline rtdetr_l_baseline    # только базовые
python evaluate.py yolo11n_improved rtdetr_l_improved    # только улучшенные
```

Ключи моделей: `yolo11n_baseline`, `rtdetr_l_baseline`, `yolo11n_improved`, `rtdetr_l_improved`.

### `custom_detector.py`

Детектор «с нуля» на PyTorch в стиле **YOLOv1-tiny** (anchor-free, single-scale, grid-based). **Архитектура:**

- Backbone: MobileNet-подобная CNN. Stem (Conv 3×3, stride 2) → 7 depthwise-separable блоков с постепенным downsample до feature-map 512 @ 7×7 (из входа 224×224).
- **Grid-голова:** один `1×1` Conv, выход `(B, 5+num_classes, 7, 7) = (B, 9, 7, 7)`. Каждая из 49 ячеек независимо предсказывает:
  - `obj` (sigmoid) — есть ли центр объекта в этой ячейке;
  - `(dx, dy)` (sigmoid) — смещение центра внутри ячейки в `[0, 1]`;
  - `(w, h)` (sigmoid) — ширина/высота рамки как доля от изображения;
  - 4 логита класса (softmax).
- Функция потерь: `BCE(obj)` на всех 49 ячейках (с разными весами `λ_obj=1.0` для "положительных" и `λ_noobj=0.5` для пустых, как в оригинальном YOLOv1) + `SmoothL1(box) × 5` и `CrossEntropy(class)` только на ячейках с объектом.
- **Инференс:** per-class greedy NMS (IoU ≥ 0.5) над всеми предсказаниями с `obj > 0.05`.
- **Метрика `mAP@0.5`:** настоящий COCO-style: предсказания сортируются по уверенности, считается precision-recall кривая, интегрируется 101-точечной интерполяцией отдельно по каждому классу, затем усредняется.

Grid используется потому что у нас **4 класса** и часто **стада** (2-5 буйволов/зебр в кадре), — single-box голова физически ограничена сверху recall'ом по числу кадров-с-одним-объектом. Grid-голова даёт возможность предсказать до 49 боксов на изображении.

Обучаются **два варианта:** без аугментации и с аугментацией (ColorJitter + случайный горизонтальный флип с согласованным отражением координат).

```bash
python custom_detector.py                 # 60 эпох на вариант (по умолчанию)
python custom_detector.py --epochs 30     # меньше эпох — быстрее, но сильнее переобучение
```

Веса: `runs/custom/{no_aug,with_aug}/best.pt`.

### `run_all.py`

Полный end-to-end прогон: подготовка датасета → baseline → оценка → improved → оценка → custom → сводная оценка. Останавливается на первой упавшей стадии.

```bash
python run_all.py
```

## 6. Результаты

Оценка на `test` split (227 изображений, seed=42). Прогон выполнен на MacBook (Apple M4, MPS, Python 3.14 + torch 2.11 + ultralytics 8.4.41).

### Базовый бейзлайн (Раздел 2)

5 эпох, `batch=4`, `imgsz=640`, гиперпараметры по умолчанию.

| Модель | Тип | P | R | mAP@0.5 | mAP@0.5:0.95 | Время обучения |
|---|---|---|---|---|---|---|
| yolo11n | CNN | 0.908 | 0.899 | 0.945 | 0.755 | ~12 мин |
| rtdetr-l | Трансформер | **0.954** | 0.903 | **0.963** | **0.825** | ~58 мин |

### Улучшенный бейзлайн (Раздел 3)

15 эпох, mosaic=1.0, mixup=0.15, HSV-джиттер, поворот ±5°, отражение, cos_lr, warmup_epochs=2, weight_decay=5e-4, label_smoothing=0.05.

| Модель | Тип | P | R | mAP@0.5 | mAP@0.5:0.95 | Δ mAP@0.5 | Δ mAP@0.5:0.95 |
|---|---|---|---|---|---|---|---|
| yolo11n | CNN | **0.963** | 0.898 | **0.969** | 0.737 | **+0.024** | −0.018 |
| rtdetr-l | Трансформер | 0.950 | **0.920** | 0.961 | 0.777 | −0.002 | −0.048 |

### Кастомный детектор (Раздел 4)

60 эпох на вариант, AdamW (lr=1e-3, weight_decay=1e-4), cosine LR, imgsz=224, grid-голова 7×7.

Два варианта (соответствуют пунктам 4a-e и 4f-j ТЗ):
- **custom_baseline** — без аугментации, чистый pipeline (пункт 4a-e).
- **custom_improved** — добавлены техники из улучшенного бейзлайна (3c) в урезанном виде, совместимом с from-scratch реализацией: ColorJitter + горизонтальный флип как аналог `hsv_*`/`fliplr`/`mosaic`/`mixup` из `train_improved.py`. AdamW + weight_decay=1e-4 + cosine LR уже используются обоими вариантами.

| Вариант | P | R | mAP@0.5 | mAP@0.5:0.95 |
|---|---|---|---|---|
| custom_baseline | 0.234 | 0.215 | 0.167 | **0.074** |
| custom_improved | **0.261** | **0.380** | **0.213** | 0.063 |

Интересно: `custom_improved` выигрывает по mAP@0.5 (+0.046), но проигрывает по mAP@0.5:0.95 (−0.011). Аугментация (ColorJitter + флип) находит больше объектов (recall 0.380 vs 0.215), но смазывает точность локализации — на строгих порогах IoU 0.55-0.95 часть совпадений теряется. Тот же эффект виден у YOLO improved (см. таблицу выше).

#### Сравнение: `custom_baseline` vs базовый бейзлайн (пункт 4d)

| Модель | mAP@0.5 | mAP@0.5:0.95 | Δ mAP@0.5 vs custom_baseline |
|---|---|---|---|
| yolo11n_baseline | 0.945 | 0.755 | **+0.778** |
| rtdetr_l_baseline | 0.963 | 0.825 | **+0.796** |
| custom_baseline | 0.167 | 0.074 | — |

#### Сравнение: `custom_improved` vs улучшенный бейзлайн (пункт 4i)

| Модель | mAP@0.5 | mAP@0.5:0.95 | Δ mAP@0.5 vs custom_improved |
|---|---|---|---|
| yolo11n_improved | 0.969 | 0.737 | **+0.756** |
| rtdetr_l_improved | 0.961 | 0.777 | **+0.748** |

Разрыв ≈0.75-0.8 mAP@0.5 между кастомной моделью и ultralytics-семейством в обоих сценариях (baseline и improved). Техники улучшенного бейзлайна (3c) сокращают отставание незначительно: `custom_improved − custom_baseline = +0.046`, тогда как `yolo11n_improved − yolo11n_baseline = +0.024` — относительный прирост у кастомной модели даже больше, но в абсолюте она по-прежнему далеко позади. Причины разобраны в выводе №3 ниже.

### Выводы

**1. Baseline: RT-DETR > YOLOv11n, но в 5× дороже.** На 5 эпохах RT-DETR-l заметно опережает yolo11n по строгой метрике mAP@0.5:0.95 (0.825 vs 0.755 — +7 п.п.) за счёт более точной локализации трансформер-головы. Ценой — ~58 мин обучения против ~12 мин у yolo11n. Для ренжер-сервиса, который чаще всего смотрит на mAP@0.5 (важен сам факт обнаружения, а не идеальная рамка), разрыв сжимается до +1.8 п.п. — выбор модели зависит от того, насколько точная локализация реально нужна.

**2. Improved конфигурация полезна только для YOLOv11n.** У yolo11n за 15 эпох mAP@0.5 вырос на +2.4 п.п. до 0.969 — агрессивные аугментации (mosaic/mixup) успели дать регуляризирующий эффект на маленькой модели. Но mAP@0.5:0.95 при этом просел на −1.8 п.п.: cильные сдвиги/повороты чуть "смазали" точность рамки. У RT-DETR-l improved практически не изменил результат (mAP@0.5: −0.2 п.п., mAP@0.5:0.95: −4.8 п.п.) — крупная предобученная модель уже в baseline близка к потолку на датасете из 1052 картинок, а сильные аугментации только мешают.

**3. Кастомный grid-детектор: recall вырос почти в 2×, но всё равно далеко до pretrained-моделей.** mAP@0.5 = 0.167–0.213 (true COCO) против 0.945+ у yolo11n. Архитектурное улучшение дало нужный эффект: grid-голова с 49 ячейками реально ловит по 2-3 объекта на кадр (recall 0.380 vs 0.208 — +82%). Но абсолютный потолок ограничен:
- **Нет предобучения** — yolo11n и rtdetr-l стартуют с весов COCO (80 классов, 330k изображений). Наша сеть инициализируется случайно и видит только 1052 картинки. Фундаментальная проблема — недостаток данных для обучения хороших низкоуровневых признаков с нуля.
- **Single-scale, single-anchor** — настоящий YOLO использует feature-pyramid (FPN) + якоря разных размеров. У нас каждая ячейка предсказывает ровно одну рамку «своего» размера. Объекты сильно разного масштаба (слон vs вдалеке зуебра) друг-другу мешают.
- **60 эпох + cosine LR без warmup** — без warmup большой lr на первых батчах разрушает начальные градиенты батчнорма; с предобучением это нормально, с нуля — хуже.

Без аугментации `custom_baseline` на 60 эпохах `train_loss` падает до 0.02 (почти идеальная подгонка train) — классический overfit. Augmentation работает как регуляризатор: `custom_improved` держит `train_loss ≈ 0.25` в конце обучения, а test-metrics ощутимо лучше (0.380 vs 0.215 recall).

**4. Аугментация критична для кастомного детектора, нейтральна-или-вредна для pretrained.** Маленькая модель без предобучения переобучается за 30-40 эпох — augmentation даёт +17 п.п. recall и +4.6 п.п. mAP@0.5. У pretrained yolo11n/rtdetr-l эффект аугментации смазан: они уже знают, как выглядят похожие животные (COCO включает классы `zebra`, `elephant`), и дополнительная вариабельность входов скорее мешает, чем помогает при коротком fine-tuning.

**5. Практические рекомендации для продакшн-сценария (мониторинг дикой природы).** Для непрерывного потока с камер-ловушек оптимальный компромисс — yolo11n improved: 0.969 mAP@0.5 при времени инференса ~8 мс на M4 (из `evaluate.py` спидометра). Если нужна лучшая локализация для подсчёта особей вплотную — rtdetr-l baseline, но инференс ~70 мс и обучение в 5× медленнее. Кастомный детектор в продакшн непригоден — учебная демонстрация пайплайна.
