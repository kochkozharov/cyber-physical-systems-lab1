# Лабораторная работа 1: Детекция объектов — African Wildlife

Вариант задания: **«на тройку»** (обнаружение и распознавание объектов, ultralytics / YOLOv11).

## 1. Датасет и обоснование выбора

**[African Wildlife Dataset](https://docs.ultralytics.com/datasets/detect/african-wildlife/)** — публичный датасет для обнаружения четырёх видов африканских животных на фотографиях: **буйвол, слон, носорог, зебра**. Архив содержит 1504 изображения уже разбитые в готовую YOLO-разметку:

| Split | Images |
|---|---|
| train | 1052 |
| val | 225 |
| test | 227 |

**Практическая задача:** автоматический мониторинг миграций и подсчёт популяции диких животных по снимкам с камер-ловушек и беспилотников в заповедниках. Это стандартная задача conservation ML — единое решение обнаруживает несколько таксонов в одной модели, что удобно для ренжер-сервисов.

Данный датасет **отсутствует** в списке уже разобранных датасетов курса и не повторяет ни один из уже занятых вариантов (в списке встречается похожее по названию `object-detection-wildlife-dataset-yolo-format` на Kaggle, но это другой набор с иными классами).

## 2. Метрики качества

| Метрика | Обоснование |
|---|---|
| **mAP@0.5** | Основная стандартная метрика детекции: площадь под кривой precision-recall при IoU ≥ 0.5. Именно она репортится в карточках моделей Ultralytics. |
| **mAP@0.5:0.95** | Строгая метрика COCO, усредняющая результаты по порогам IoU от 0.5 до 0.95 с шагом 0.05. Штрафует за неточную локализацию рамки. |
| **Precision** | Доля предсказанных рамок, которые верны. Важна, чтобы ренжеры не тратили время на ложные срабатывания. |
| **Recall** | Доля реальных животных, которые были найдены. Важна, чтобы не пропустить особь при учёте популяции. |

Для кастомного детектора (раздел 4) mAP@0.5 считается как доля изображений, где единственное предсказание модели совпало хотя бы с одной GT-рамкой (IoU ≥ 0.5 + верный класс).

## 3. Установка

Требуется **Python 3.12+** (проверено на Python 3.14, macOS arm64 / MPS). Для apple silicon PyTorch использует MPS; RT-DETR падает на некоторых оп-ах обратно на CPU — скрипты выставляют `PYTORCH_ENABLE_MPS_FALLBACK=1` автоматически.

```bash
python3 -m venv .venv
source .venv/bin/activate     # bash / zsh
#   или:  source .venv/bin/activate.fish
pip install -r requirements.txt
```

Датасет **скачивается автоматически** при первом запуске `prepare_dataset.py` — никаких внешних манипуляций с Kaggle/Roboflow не требуется. Архив (~105 MB) берётся из официального GitHub-релиза Ultralytics.

## 4. Скрипты

### `prepare_dataset.py`

Скачивает архив `african-wildlife.zip` из Ultralytics assets, распаковывает его в `dataset/african-wildlife/` (`images/{train,val,test}` + `labels/{train,val,test}`) и записывает `data/dataset.yaml` с абсолютными путями.

```bash
python prepare_dataset.py
```

Ожидаемый вывод:

```
Downloading dataset from https://github.com/ultralytics/assets/releases/download/v0.0.0/african-wildlife.zip
Extracting into .../dataset
Dataset prepared at .../dataset/african-wildlife
Wrote config to .../data/dataset.yaml
Dataset splits: 1052 train / 225 val / 227 test
```

### `train_baseline.py`

Обучает две базовые модели из семейства Ultralytics:

- **yolo11n** — YOLOv11 nano, чисто свёрточный детектор (~2.6 M параметров).
- **rtdetr-l** — RT-DETR large, трансформер-encoder/decoder (~32 M параметров).

Обе модели — **5 эпох, batch=16, imgsz=640, seed=42**, все остальные гиперпараметры дефолтные.

```bash
python train_baseline.py
```

Веса: `runs/baseline/{yolo11n,rtdetr_l}/weights/best.pt`.

### `train_improved.py`

Переобучает обе модели с улучшенной конфигурацией, проверяющей три гипотезы:

| Гипотеза | Изменение | Ожидаемый эффект |
|---|---|---|
| Более сильная аугментация | mosaic=1.0, mixup=0.15, HSV-джиттер, поворот ±5°, отражение | Лучшая обобщающая способность |
| Косинусный LR + прогрев | cos_lr=True, warmup_epochs=2 | Более стабильная сходимость |
| Регуляризация | weight_decay=5e-4, label_smoothing=0.05 | Снижение переобучения |

Бюджет увеличен до **15 эпох**, чтобы косинусный график и аугментации успели реально повлиять — на 5 эпохах warmup занимал бы 40% обучения.

```bash
python train_improved.py
```

Веса: `runs/improved/{yolo11n,rtdetr_l}/weights/best.pt`.

### `evaluate.py`

Оценивает сохранённые веса на **тестовой выборке** (`split=test`) и печатает markdown-таблицу с Precision / Recall / mAP@0.5 / mAP@0.5:0.95.

```bash
python evaluate.py                                        # все доступные модели
python evaluate.py yolo11n_baseline rtdetr_l_baseline    # только базовые
python evaluate.py yolo11n_improved rtdetr_l_improved    # только улучшенные
```

Ключи моделей: `yolo11n_baseline`, `rtdetr_l_baseline`, `yolo11n_improved`, `rtdetr_l_improved`.

### `custom_detector.py`

Детектор «с нуля» на PyTorch. **Архитектура:**

- Backbone: MobileNet-подобная CNN. Stem (Conv 3×3, stride 2) → 6 depthwise-separable блоков с постепенным downsample → Global Average Pooling → 512-мерный вектор.
- Две головы (каждая — двухслойный MLP): регрессия рамки `(cx, cy, w, h)` (Sigmoid) + классификация на 4 класса (CrossEntropy).
- Функция потерь: `SmoothL1(box) + CrossEntropy(class)`.
- **Упрощение:** одно предсказание на изображение. Во время обучения в качестве таргета берётся наибольшая по площади рамка; во время оценки предсказание матчится с *любой* GT-рамкой на изображении (IoU ≥ 0.5 + верный класс).

Обучаются **два варианта:** без аугментации и с аугментацией (ColorJitter + случайный горизонтальный флип с согласованным отражением координат).

```bash
python custom_detector.py                 # 10 эпох на вариант
python custom_detector.py --epochs 20     # переопределить бюджет
```

Веса: `runs/custom/{no_aug,with_aug}/best.pt`.

### `run_all.py`

Полный end-to-end прогон: подготовка датасета → baseline → оценка → improved → оценка → custom → сводная оценка. Останавливается на первой упавшей стадии.

```bash
python run_all.py
```


## 6. Результаты

Оценка на `test` split (227 изображений, seed=42). Прогон выполнен на MacBook (Apple M4, MPS, Python 3.14 + torch 2.11 + ultralytics 8.4.41).

### Базовый бейзлайн (Раздел 2)

5 эпох, `batch=4`, `imgsz=640`, гиперпараметры по умолчанию.

| Модель | Тип | P | R | mAP@0.5 | mAP@0.5:0.95 | Время обучения |
|---|---|---|---|---|---|---|
| yolo11n | CNN | 0.908 | 0.899 | 0.945 | 0.755 | ~12 мин |
| rtdetr-l | Трансформер | **0.954** | 0.903 | **0.963** | **0.825** | ~58 мин |

### Улучшенный бейзлайн (Раздел 3)

15 эпох, mosaic=1.0, mixup=0.15, HSV-джиттер, поворот ±5°, отражение, cos_lr, warmup_epochs=2, weight_decay=5e-4, label_smoothing=0.05.

| Модель | Тип | P | R | mAP@0.5 | mAP@0.5:0.95 | Δ mAP@0.5 | Δ mAP@0.5:0.95 |
|---|---|---|---|---|---|---|---|
| yolo11n | CNN | **0.963** | 0.898 | **0.969** | 0.737 | **+0.024** | −0.018 |
| rtdetr-l | Трансформер | 0.950 | **0.920** | 0.961 | 0.777 | −0.002 | −0.048 |

### Кастомный детектор (Раздел 4)

10 эпох на вариант, Adam, lr=1e-3, imgsz=224, MobileNet-подобный backbone → MLP-головы (одно предсказание на изображение).

| Вариант | P | R | mAP@0.5 |
|---|---|---|---|
| custom (без аугментации) | 0.278 | 0.168 | 0.278 |
| custom (с аугментацией) | **0.344** | **0.208** | **0.344** |

### Выводы

**1. Baseline: RT-DETR > YOLOv11n, но в 5× дороже.** На 5 эпохах RT-DETR-l заметно опережает yolo11n по строгой метрике mAP@0.5:0.95 (0.825 vs 0.755 — +7 п.п.) за счёт более точной локализации трансформер-головы. Ценой — ~58 мин обучения против ~12 мин у yolo11n. Для ренжер-сервиса, который чаще всего смотрит на mAP@0.5 (важен сам факт обнаружения, а не идеальная рамка), разрыв сжимается до +1.8 п.п. — выбор модели зависит от того, насколько точная локализация реально нужна.

**2. Improved конфигурация полезна только для YOLOv11n.** У yolo11n за 15 эпох mAP@0.5 вырос на +2.4 п.п. до 0.969 — агрессивные аугментации (mosaic/mixup) успели дать регуляризирующий эффект на маленькой модели. Но mAP@0.5:0.95 при этом просел на −1.8 п.п.: cильные сдвиги/повороты чуть "смазали" точность рамки. У RT-DETR-l improved практически не изменил результат (mAP@0.5: −0.2 п.п., mAP@0.5:0.95: −4.8 п.п.) — крупная предобученная модель уже в baseline близка к потолку на датасете из 1052 картинок, а сильные аугментации только мешают.

**3. Кастомный детектор сильно проигрывает YOLO.** mAP@0.5 = 0.278–0.344 против 0.945+ у yolo11n. Причины:
- **Архитектурное упрощение** — одна рамка на изображение. В African Wildlife часто 2-5 животных в кадре (например, стадо буйволов), и head физически не может их все найти: recall ограничен сверху долей кадров, где одна самая крупная рамка — это и есть "правильный ответ".
- **4 класса вместо 1** — задача сложнее, чем в `README_example.md` (где был один класс `animal`). Классификационная голова теряет ~30% предсказаний именно на классе, а не на локализации.
- **Нет предобучения** — yolo11n и rtdetr-l стартуют с весов COCO, кастомный — с нуля за 10 эпох.

**4. Аугментация помогла только кастомному детектору.** +6.6 п.п. mAP@0.5 (0.278 → 0.344) — классический эффект регуляризации маленькой модели, которая без аугментации успевает переобучиться за 10 эпох. У больших pretrained-моделей эффект обратный: аугментация мешает точной локализации.

**5. Практические рекомендации для продакшн-сценария (мониторинг дикой природы).** Для непрерывного потока с камер-ловушек оптимальный компромисс — yolo11n improved: 0.969 mAP@0.5 при времени инференса ~8 мс на M4 (из `evaluate.py` спидометра). Если нужна лучшая локализация для подсчёта особей вплотную — rtdetr-l baseline, но инференс ~70 мс и обучение в 5× медленнее. Кастомный детектор в продакшн непригоден — учебная демонстрация пайплайна.
